# 任务分辨率阶梯定价兼容性说明

## 1. 文档目的

本文记录任务分辨率阶梯定价上线后仍然保留的旧架构兼容行为，作为后续维护、发布、回滚和删除兼容代码时的依据。

当前兼容策略的目标是：新配置统一向分辨率矩阵演进，同时避免旧价格配置、历史任务、外部 API 客户端和旧版 Playground 在升级时立即失效。兼容层不是两套定价架构长期并行演进的承诺。

适用范围：

- `billing_setting.task_pricing` 中的任务价格配置；
- 任务报价、消费和退款快照；
- `/api/pricing` 与 `/api/user/models`；
- 管理端价格编辑器、公开价格页和 Playground；
- 多实例滚动升级与版本回滚。

---

## 2. 新旧配置结构

### 2.1 旧平铺结构

旧配置对上游支持的所有分辨率使用同一组价格：

```json
{
  "AP Seedance-2.0 标准版": {
    "unit": "second",
    "no_reference_video_unit_price": 0.04,
    "reference_video_policy": "custom",
    "reference_video_unit_price": 0.06
  }
}
```

### 2.2 新矩阵结构

新配置按上游目录返回的规范分辨率键分别定价：

```json
{
  "AP Seedance-2.0 标准版": {
    "unit": "second",
    "by_resolution": {
      "480p": {
        "no_reference_video_unit_price": 0.04,
        "reference_video_policy": "custom",
        "reference_video_unit_price": 0.06
      },
      "720p": {
        "no_reference_video_unit_price": 0.08,
        "reference_video_policy": "same"
      },
      "4k": {
        "no_reference_video_unit_price": 0.5,
        "reference_video_policy": "disabled"
      }
    }
  }
}
```

### 2.3 严格边界

- 一个模型只能使用旧平铺结构或新矩阵结构，不能混用。
- 旧平铺结构会把同一价格应用到当前上游支持的全部分辨率。
- 矩阵结构必须严格命中所选分辨率；缺少档位时返回 `resolution_price_not_configured`，不得回退到旧平铺价格或其他档位。
- 分辨率只执行 `trim + lowercase` 标准化，不把 `2k` 与 `1440p`、`4k` 与 `2160p` 自动互转。
- 上游目录不支持请求分辨率时返回 `unsupported_resolution`。
- 本地配置与上游能力没有任何有效交集时，该模型不会出现在公开价格和 Playground 有效模型列表中。

## 3. 当前兼容项清单

| 编号 | 兼容项 | 当前行为 | 保留级别 | 删除前提 |
|---|---|---|---|---|
| C01 | 旧平铺价格配置 | 后端继续解析、校验和报价；管理端读取或普通保存时不会自动转换。使用部署技能并明确授权“覆盖 AIPDD 模型价格”时，AIPDD 旧平铺配置会由认证目录重建为矩阵 | 中期 | 所有存量配置已迁移为矩阵，并至少经过一个稳定发布周期 |
| C02 | 管理端显式转换 | 旧配置默认保持旧结构，只有管理员主动切换到矩阵时才转换 | 中期 | C01 删除时一并移除旧结构编辑入口 |
| C03 | 历史任务缺少 `resolution` | `TaskPricingQuote`、`TaskBillingContext` 和 AIPDD 快照中的 `resolution` 均为可选字段，旧 JSON 可以继续读取和退款 | 长期 | 原则上不删除；只有历史任务已全部过期且有离线归档迁移方案时才能评估 |
| C04 | `/api/user/models` 旧响应 | 默认继续返回模型名称字符串数组；只有 `?details=true` 返回模型及 `video_resolutions` | 中期或长期 | 已发布新的 API 版本，且确认没有外部客户端依赖旧响应 |
| C05 | `/api/pricing` 缺少能力字段 | 新响应增加 `task_pricing_resolutions`；前端仍能读取旧后端未提供该字段的响应 | 中期 | 不再支持新前端连接旧后端，且滚动升级窗口已经结束 |
| C06 | Playground 硬编码分辨率 | 优先使用 `details=true` 返回的有效分辨率；能力元数据缺失时退回旧硬编码列表 | 短期 | 所有受支持后端都稳定提供能力元数据，并经过监控确认无回退使用 |
| C07 | 旧任务退款额度 | 退款使用任务创建时冻结的原始额度，不重新读取实时矩阵价格 | 长期 | 不应删除，这是账单一致性要求，不只是迁移兼容 |
| C08 | 旧版回滚 | 新版本不修改数据库结构；但旧程序不能正确管理新矩阵配置，回滚时必须恢复备份的旧平铺配置 | 运维兼容 | 矩阵成为唯一格式且明确不再支持回滚到旧版本 |

## 4. 必须长期保持的账单兼容

历史任务可能没有以下新字段：

```json
{
  "resolution": "720p"
}
```

读取旧任务时，字段缺失应解释为“历史记录未保存该事实”，不能解释为当前请求的默认分辨率，也不能据此重新报价。

退款与审计必须遵守以下规则：

1. 使用首次报价冻结的额度、单价、秒数和参考视频状态。
2. 新任务同时冻结分辨率；旧任务允许没有分辨率。
3. 渠道重试复用首次报价，不按重试时的实时价格重新计算。
4. 失败退款按原始冻结额度全额退还，不读取当前任务价格配置或 AIPDD 目录价格。
5. 不应为了补齐旧数据而猜测 `resolution`，也不应根据宽高反向修改历史账单事实。

这部分兼容关系到账单可追溯性，建议长期保留。

## 5. API 兼容约定

### 5.1 用户模型接口

旧调用：

```http
GET /api/user/models
```

继续返回模型名称字符串数组，保持现有客户端兼容。

需要视频能力详情的新调用：

```http
GET /api/user/models?details=true
```

返回示例：

```json
{
  "success": true,
  "data": [
    {
      "model": "AP Seedance-2.0 标准版",
      "video_resolutions": ["480p", "720p", "1080p", "4k"]
    }
  ]
}
```

不要在现有默认接口上直接把字符串数组替换为对象数组。若未来需要删除旧响应，应通过明确的 API 版本升级完成。

### 5.2 公开价格接口

`/api/pricing` 中的 `task_pricing_resolutions` 表示“上游当前支持且本地已经定价”的有效交集，不是本地保存的全部预配置档位。

- 字段缺失：可能是旧后端响应，前端可以使用旧兼容展示。
- 空数组：新后端明确表示当前没有有效分辨率，不能当成“所有分辨率都有效”。
- 非空数组：公开起价、排序和详情矩阵只能遍历这些有效档位。

## 6. 管理端和 Playground 兼容

### 6.1 管理端

- 读取旧平铺配置时保持原结构，不自动写成矩阵。
- 新建结构化任务价格时优先使用矩阵。
- 管理员可以保存上游目录暂未出现的规范档位，界面标记为“未激活”。
- JSON 与可视化编辑、批量复制必须深拷贝并完整保留嵌套和未知字段。
- 管理员从旧结构切换到矩阵属于显式迁移操作，保存前应核对每个分辨率的价格和视频策略。

### 6.2 Playground

Playground 的分辨率来源顺序为：

1. `/api/user/models?details=true` 返回的当前有效分辨率；
2. 能力元数据缺失时，使用已有硬编码列表作为临时回退。

硬编码列表不能作为新增 AIPDD 档位的事实源。例如 `2k` 只有在 AIPDD 目录返回规范键且本地矩阵已配置价格后才应自动出现。

## 7. 发布和回滚要求

### 7.1 首次启用矩阵前

1. 备份 `billing_setting.task_pricing`。
2. 完成所有 NewAPI 实例升级。
3. 刷新管理后台页面，确保管理员使用新前端。
4. 再保存第一个矩阵配置。

旧实例或旧管理端可能不认识 `by_resolution`，因此多实例环境不能在新旧版本混跑期间写入矩阵配置。

### 7.2 回滚旧版本

代码回滚到不支持矩阵的版本时，必须同时恢复升级前备份的旧平铺配置。只回滚程序、不恢复配置不属于受支持的回滚路径。

本功能没有数据库迁移，但“没有数据库迁移”不代表新配置 JSON 能被旧程序安全编辑。

## 8. 建议的兼容层退出路线

### 阶段一：当前版本

- 双结构读取和编辑。
- 新配置默认使用矩阵。
- 保留旧 API 响应和 Playground 回退。
- 在管理端标识旧结构和未激活档位。

### 阶段二：迁移版本

- 提供一键把旧平铺价格展开为各有效分辨率矩阵的能力。
- 转换前展示差异并要求管理员确认。
- 使用 `new-api-docker-deploy` 更新时，明确授权覆盖 AIPDD 模型价格也可完成 AIPDD 旧平铺配置到目录矩阵的重建；拒绝覆盖时必须原样保留。
- 记录仍在使用旧结构的模型数量，以及 Playground 能力回退是否仍被触发。
- 停止新建旧平铺配置，但后端继续读取已有配置。

### 阶段三：矩阵唯一写入

- 管理端只写矩阵结构。
- 后端继续只读兼容旧结构一个稳定周期。
- 发布说明明确旧结构的最终删除版本。

### 阶段四：删除旧定价分支

只有同时满足以下条件后，才删除旧平铺报价逻辑：

- 存量 `task_pricing` 中没有旧平铺配置；
- 所有运行实例和管理端均支持矩阵；
- 已经过至少一个完整稳定发布周期；
- 有可验证的配置备份和恢复方案；
- 旧 API 客户端影响已经单独评估，不与配置格式删除混为一次变更。

即使进入阶段四，也应继续保留历史任务缺少 `resolution` 的读取和退款兼容。

## 9. 修改兼容代码时的测试要求

涉及上述兼容层的修改至少应覆盖：

- 旧平铺配置解析、报价和管理端往返不变形；
- 新旧字段混用时拒绝保存；
- 矩阵缺档时不回退；
- 历史任务没有 `resolution` 时正常读取和退款；
- 默认 `/api/user/models` 仍返回字符串数组；
- `details=true` 返回有效分辨率；
- `/api/pricing` 区分字段缺失、空数组和非空数组；
- Playground 能力优先及旧数据回退；
- 滚动发布期间旧前端不会覆盖新矩阵配置；
- 失败退款和渠道重试始终使用冻结报价。

建议验收命令：

```text
go test ./...
cd web/default
bun test
bun run typecheck
bun run build
bun run i18n:sync
```

## 10. 主要实现位置

- `setting/billing_setting/task_pricing.go`：新旧配置结构、校验、深拷贝和报价。
- `relay/relay_task.go`：计费事实、首次报价冻结和重试复用。
- `relay/channel/task/aipdd/adaptor.go`：AIPDD 分辨率能力、默认时长/FPS 和 remix 继承。
- `model/task.go`：任务计费上下文与 AIPDD 快照。
- `model/pricing.go`：公开价格与有效分辨率交集。
- `controller/user.go`：用户模型旧响应和 `details=true` 模式。
- `web/default/src/features/system-settings/models/model-pricing-sheet.tsx`：旧结构编辑和矩阵显式迁移。
- `web/default/src/features/playground/constants.ts`：动态能力优先与硬编码回退。
